# GPT-SELF — Система памяти и блоков (Спецификация v1.0)

> **Этот файл описывает архитектуру блоковой памяти.**
> Основной мастер-документ: GPT_SELF_MASTER.md

**Дата:** 2026-03-11
**Версия:** 1.0

-----

## 1. КОНЦЕПЦИЯ

Любая запись пользователя (шаги, благодарности, SOS, чувства, профиль, свободный рассказ)
автоматически раскладывается по **9 корневым блокам**, формируя карту личности.

Два уровня маршрутизации:

1. **Словарь лемм** (pymorphy2 → леммы → lexicon_ru.yaml → block/subblock) — покрывает 70-80% случаев
1. **Мини-LLM классификатор** (GPT-4o mini, строгий JSON) — fallback когда словарь не справился

-----

## 2. ОСНОВНЫЕ ПОНЯТИЯ

### 2.1 Блок

Верхний контейнер по крупной теме жизни. Имеет:

- человекочитаемое название
- тип (profile / functional / recovery / hidden)
- правила видимости и хранения

### 2.2 Подблок

Уточняющий контейнер внутри блока: «Люди → Семья → Сестра».
Создаются:

- A) вручную пользователем
- B) автоматически при появлении новой сущности

### 2.3 Запись (Record)

Атомарная единица хранения с метаданными:

- источник (steps / gratitude / sos / profile / free_note / feelings / self_analysis)
- дата/время
- ссылка на блок/подблок
- уровень памяти (Volatile / Dynamic / Stable)
- флаги: подтверждено, чувствительное, архив
- извлечённые признаки: эмоции, ключевые фразы, сущности

-----

## 3. ТРИ УРОВНЯ ПАМЯТИ

|Уровень     |Описание                                 |Срок жизни |Пример               |
|------------|-----------------------------------------|-----------|---------------------|
|**Volatile**|Текущий диалог, оперативные детали       |Минуты/часы|«сейчас тревожно»    |
|**Dynamic** |Повторяющиеся эпизоды, активные темы     |Дни/недели |«ссора с женой вчера»|
|**Stable**  |Ценности, устойчивые факты, ядро личности|Месяцы/годы|«у меня есть дочь»   |

### Правила сохранения в Stable:

- Только по явному подтверждению пользователя
- Или безопасный технический факт (текущий шаг, факт благодарности)
- Эмоции = состояния, НЕ факты (хранятся в блоке States)
- Чувствительные данные → ВСЕГДА спросить «Сохранить в профиль?»

### Подблоки people:

- `people.family` — мама, папа, сестра, брат, жена, муж, дочь, сын…
- `people.friends` — друг, подруга, товарищ, знакомый…
- `people.work` — коллега, начальник, клиент, партнёр…
- `people.support` — психолог, спонсор, группа, сообщество…

### Правило автоподблока:

Если в тексте найден новый человек → создать подблок внутри подходящей категории.
Пример: «сестра» впервые → `people.family.sister`

### Правило разнесения денег:

- «деньги в быту» → home (бюджет, долги, коммуналка)
- «деньги в работе» → work (выручка, маржа, сделки)

## 5. ГИБРИДНАЯ МАРШРУТИЗАЦИЯ

### 5.1 Пайплайн на каждый вход

```
Текст пользователя
        ↓
[Шаг 1] Определить источник (steps/sos/gratitude/profile/...)
        ↓
[Шаг 2] pymorphy2 → леммы → поиск в lexicon_ru.yaml
        ↓
[Шаг 3] Если уверенное совпадение → primary_block из словаря
         Если конфликт/нет совпадений → мини-LLM классификатор
        ↓
[Шаг 4] Если подблока нет → создать (auto_created=true)
        ↓
[Шаг 5] Сохранить record + record_links + метаданные
        ↓
[Шаг 6] Если unknown леммы → инкремент unknown_lemmas
```

### 5.2 Маршрутизация по источнику (базовый primary)

|Источник     |Базовый primary_block     |
|-------------|--------------------------|
|steps        |recovery                  |
|self_analysis|recovery                  |
|feelings     |states                    |
|sos          |states + related по смыслу|
|gratitude    |определить по контенту    |
|profile      |block по разделу профиля  |
|free_note    |определить по контенту    |

### 5.3 Одна запись → несколько блоков

Текст «Поссорился с сестрой, тревога» содержит:

- primary: `relations` (конфликт)
- related: `people.family` (сестра) + `states` (тревога)

Запись хранится 1 раз, связи через `record_links`.

## 6. СЛОВАРЬ ЛЕММ (lexicon_ru.yaml)

### Формат записи:

```yaml
сестра:
  entity_type: person
  block: people
  subblock: family
  sensitivity: low

дом:
  entity_type: asset
  block: home
  subblock: housing
  sensitivity: high  # факт владения → подтверждение

тревога:
  entity_type: emotion
  block: states
  subblock: anxiety
  sensitivity: low
```

### Стартовый объём: ~300-400 лемм по 9 блокам

Полный список лемм — см. конец этого документа (Приложение А).

## 7. МИНИ-LLM КЛАССИФИКАТОР

### Когда вызывается:

- Словарь не сработал (нет совпадений)
- Конфликт блоков (совпали разные)
- Нужно определить primary при неоднозначности

### Модель: GPT-4o mini

### Формат ответа (строгий JSON):

```json
{
  "primary_block": "people",
  "subblock": "family",
  "entities": [
    {"type": "person", "lemma": "сестра", "name_raw": "сестру"}
  ],
  "related_blocks": ["states"],
  "memory_candidate": "dynamic",
  "needs_confirm": false,
  "confidence": 0.82,
  "reason": "в тексте упомянут родственник"
}
```

### Защита от галлюцинаций:

- `primary_block` — ТОЛЬКО из списка 9 блоков
- Валидация через pydantic-схему
- Если JSON невалиден → fallback на `unclassified`
- Если confidence < 0.65 → спросить пользователя кнопками

## 8. ПРАВИЛА ПОДТВЕРЖДЕНИЯ

### Спрашиваем пользователя если:

|Условие                               |Действие                                   |
|--------------------------------------|-------------------------------------------|
|Stable-факт из благодарности/SOS      |«Сохранить в профиль?»                     |
|Новая сущность-человек                |Создать подблок автоматически (без вопроса)|
|confidence < 0.65                     |«Куда отнести?» + кнопки блоков            |
|Чувствительная тема (sensitivity=high)|«Сохранить? Куда именно?»                  |

### НЕ спрашиваем:

- Dynamic-записи с confidence > 0.8
- Технические факты процесса (текущий шаг, дата)
- Эмоции/состояния (всегда → states)

## 9. МОДЕЛЬ БД

### Таблица user_blocks

```sql
CREATE TABLE user_blocks (
    id SERIAL PRIMARY KEY,
    user_id INTEGER REFERENCES users(id),
    parent_id INTEGER REFERENCES user_blocks(id),
    title VARCHAR(100),
    slug VARCHAR(50),
    type VARCHAR(20),  -- profile/functional/recovery/hidden
    visibility VARCHAR(20),  -- user_visible/hidden
    auto_created BOOLEAN DEFAULT false,
    created_at TIMESTAMP DEFAULT now(),
    updated_at TIMESTAMP DEFAULT now()
);
```

### Таблица user_records

```sql
CREATE TABLE user_records (
    id SERIAL PRIMARY KEY,
    user_id INTEGER REFERENCES users(id),
    source_module VARCHAR(30),  -- steps/gratitude/sos/profile/free_note/feelings/self_analysis
    content_encrypted TEXT,
    created_at TIMESTAMP DEFAULT now(),
    memory_level VARCHAR(10),  -- volatile/dynamic/stable
    confirmed_by_user BOOLEAN DEFAULT false,
    sensitive_flag BOOLEAN DEFAULT false,
    extracted_json JSONB
);
```

### Таблица record_links

```sql
CREATE TABLE record_links (
    id SERIAL PRIMARY KEY,
    record_id INTEGER REFERENCES user_records(id),
    block_id INTEGER REFERENCES user_blocks(id),
    link_type VARCHAR(10)  -- primary/related
);
```

### Таблица unknown_lemmas (телеметрия)

```sql
CREATE TABLE unknown_lemmas (
    id SERIAL PRIMARY KEY,
    lemma_hash VARCHAR(64),
    count INTEGER DEFAULT 1,
    suggested_block VARCHAR(30),
    first_seen TIMESTAMP DEFAULT now(),
    last_seen TIMESTAMP DEFAULT now()
);
```

## 10. ПОПОЛНЕНИЕ СЛОВАРЯ

### Механизм 1: User-confirm routing

- Юзер нажимает кнопку «Куда отнести?» → сохраняем выбор
- После 5+ подтверждений одной леммы → добавляем в lexicon_ru.yaml

### Механизм 2: Unknown-lemmas telemetry

- Считаем неизвестные леммы (хеш, не текст!)
- Раз в неделю: топ-30 → администратор добавляет в YAML
- Команда бота: `/lemmas_top` — показать топ-20 за неделю

### Логирование (trace_id):

Каждая классификация логируется:

- trace_id
- путь (dictionary / llm / user_confirm)
- результат (block/subblock)
- confidence
- время обработки
