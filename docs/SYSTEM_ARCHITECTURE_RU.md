# GPT-SELF — System Architecture (RU)

## 0) Назначение документа
Этот документ нужен как **карта системы**: что уже реализовано в коде, что реализовано частично, и что является целевой архитектурой.

Статусы в тексте:
- ✅ **Implemented** — подтверждено текущим кодом.
- 🟡 **Partial** — есть частично/не везде/не в целевом виде.
- 🔜 **Target** — целевая архитектура, ещё не реализована полностью.

---

## 1) Компоненты системы
- PostgreSQL (данные пользователей и прогресс)
- Backend API (FastAPI)
- Telegram Bot (aiogram)
- ChromaDB (векторное хранилище)

Статус: ✅ Implemented.

---

## 2) Пользовательский путь высокого уровня
1. Пользователь взаимодействует с ботом в Telegram.
2. Бот вызывает backend API.
3. Backend выполняет сценарий (шаги, профиль, SOS, трекер, чат).
4. Ответ возвращается в бот.

Статус: ✅ Implemented.

---

## 3) Ключевые продуктовые сценарии
- Работа по шагам + прогресс
- SOS режимы
- Благодарности + история
- Чувства + история
- 10 шаг (самоанализ)
- Профиль + секции + история

Статус: ✅ Implemented (по базовому MVP), 🟡 Partial (качество аналитики/объяснимость/наблюдаемость).

---

## 4) Персонализация и память
Система использует:
- профиль пользователя,
- историю сообщений,
- фреймы,
- векторный поиск по пользовательским событиям и ядру.

Статус: ✅ Implemented.

Ограничение продукта:
- устойчивые факты должны быть подтверждаемыми пользователем.

Статус: 🟡 Partial (правило зафиксировано, но требуется единый механизм подтверждения фактов).

---

## 5) LLM-слой
Требование архитектуры:
- единая точка вызова моделей (LLM Gateway).

Статус: 🟡 Partial.

Комментарий:
- Провайдерный слой присутствует,
- но в части бизнес-кода остаются прямые вызовы клиента модели.

---

## 6) Taxonomy / Mapping / Lexicon
Целевая модель:
- 9 корневых блоков: `identity, people, relations, health, work, home, growth, values, states`.
- legacy-термины маппятся в девятку.
- lexicon-first (lemma -> block/subblock/entity_type/sensitivity), затем fallback в mini-LLM.

Статус: 🔜 Target.

Комментарий:
- Требование зафиксировано в AGENTS,
- но требуется формализация в коде и data-contract.

---

## 7) Event index (source/primary/related)
Цель:
- нормализованный индекс событий с полями:
  - source
  - primary
  - related
  - entity/subblock
  - confidence
  - memory_level
  - needs_confirm/confirmed

Статус: 🔜 Target.

---

## 8) State Engine
Цель:
- выделенный модуль для состояний/эмоций с временной динамикой и связями с контекстом.

Статус: 🔜 Target.

---

## 9) Наблюдаемость
Цель:
- сквозной `trace_id` на действие,
- структурные логи,
- debug endpoints для объяснимости маршрута: классификация → mapping → сохранение → сбор prompt.

Статус: 🟡 Partial.

---

## 10) Принцип безопасности фактов
- не галлюцинировать факты о пользователе,
- конфликт фактов переводить в candidate,
- подтверждение изменений — через пользователя,
- версионность фактов: confirmed / candidate / archived.

Статус: 🔜 Target (часть практик есть, единого контура пока нет).

---

## 11) Smoke checklist после изменений
1. шаг → ответ → мой прогресс
2. SOS «просто тяжело»
3. SOS «не понял вопрос»
4. SOS «нужны примеры»
5. благодарность → история
6. чувства → история
7. 10 шаг → сохранение

Статус: ✅ Использовать при каждом патче.
