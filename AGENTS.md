AGENTS.md — GPT-SELF (Project Context + Rules for Codex)

0) Продуктовый смысл (зачем система существует)
GPT-SELF Recovery — это не «чат ради чата», а ежедневная опора процесса восстановления в 12-шаговой программе.
Система должна помогать пользователю не терять нить Recovery в реальной жизни: фиксировать след процесса,
показывать динамику за 7/30 дней и поддерживать возвращение в путь без давления и морализаторства.

Ожидаемый результат для пользователя:
- держать непрерывность восстановления шаг за шагом;
- видеть повторяющиеся паттерны и триггеры;
- получать понятные срезы «что меняется во мне»;
- иметь поддержку в трудные моменты (особенно в SOS) без перегруза анализом.

Базовые продуктовые ограничения:
- не галлюцинировать факты о человеке;
- устойчивые факты профиля и изменения ядра — только управляемо и с подтверждением пользователя;
- эмоции фиксируются как состояния, привязанные к контексту/событиям, а не как «факты о мире».

1) Что это за проект (коротко)
GPT-SELF — Telegram-бот + backend (FastAPI), который помогает человеку держать процесс Recovery (12 шагов)
в обычной жизни. Это не “чат ради чата”, а структура процесса: шаги, 10 шаг (самоанализ), чувства,
благодарности, SOS, профиль, прогресс и история.

2) Главная цель на текущем этапе
Довести MVP Recovery до прод-стабильности и запустить монетизацию:
- стабильность сохранений и навигации
- наблюдаемость (trace/debug), чтобы понимать “что где и почему”
- оплата/подписки
- шифрование (после trace/debug, иначе ослепнем)

4) Обязательные принципы архитектуры (DO)
A) Единая точка вызова моделей:
Никаких прямых вызовов OpenAI/Anthropic в бизнес-коде.
Все вызовы LLM проходят только через LLM Gateway (единый модуль).
Цель: менять провайдера/модель без переписывания, управлять лимитами и логами.

B) Девятка корневых блоков (taxonomy):
identity, people, relations, health, work, home, growth, values, states
Любая запись должна иметь primary (один главный блок) и может иметь related (0..N).

C) Mapping:
Если текущий LLM-классификатор возвращает старые названия (“семья/финансы/духовность”),
они маппятся в девятку.

D) Lexicon до LLM:
Добавляем lexicon_ru.yaml (lemma -> block/subblock/entity_type/sensitivity).
Сначала пробуем словарь лемм, только если нет/конфликт — mini-LLM fallback (строгий JSON).

E) source + primary + related:
Для общего индекс-слоя (event_index):
- source (steps/gratitude/sos/profile/feelings/step10)
- primary (из девятки)
- related (из девятки)
- entity/subblock (например people.family.sister)
- confidence, memory_level, needs_confirm/confirmed

F) Stable-факты не переписывать автоматически:
Конфликт (“сестра Аня” vs “сестра Оля”) → candidate → спросить пользователя
(вторая сестра / исправление / не сохранять). Факты версионно: confirmed/candidate/archived.

G) State Engine (эмоции/состояния) отдельным модулем:
StateEvent: emotion + timestamp + source + confidence + link to entity/block.
Срезы: current_state + 7/30 дней + связи “сущность ↔ эмоция”.

H) Observability:
trace_id на каждое действие + структурные логи + debug endpoints.
Нужно видеть: классификацию, mapping, куда сохранили, из чего собран prompt.

I) Обязательные продуктовые способности (в терминах поведения системы):
1. Видеть прогресс:
   - шаги (пройденные вопросы/этапы),
   - регулярность (ответы/благодарности/самоанализ),
   - динамика 7/30 дней.
2. Видеть эмоциональную динамику:
   - преобладающие чувства/состояния,
   - изменения во времени,
   - повторяющиеся состояния (например, тревога/злость/пустота/перегруз/тяга).
3. Видеть связи контекст → состояние:
   - например «семья → папа → тревога», «работа → перегруз», «изоляция → пустота».
4. Поддерживать шаговую работу:
   - «не понял вопрос» → объяснить смысл,
   - «нужны примеры» → дать примеры,
   - «просто тяжело» → поддержка без анализа.
5. Делать срезы истории:
   - что повторялось,
   - точки роста/провалов,
   - что помогает возвращаться,
   - какие паттерны включаются.

5) Что нельзя ломать (DO NOT BREAK)
- шаг → ответ → мой прогресс
- SOS кнопки: “просто тяжело”, “не понял вопрос”, “нужны примеры”, “куда смотреть”
- благодарности → история
- чувства → история
- 10 шаг → сохранение
- профиль → секции + история
- миграции Alembic / DATABASE_URL
- /health на backend

6) Требования к любому патчу/PR
- Маленькие итерации.
- Список изменённых файлов.
- Инструкция “как проверить” (smoke checklist).
- Не коммитить секреты (.env и ключи).

7) Smoke checklist (после каждого изменения)
1) шаг → ответ → мой прогресс
2) SOS “просто тяжело” (без анализа)
3) SOS “не понял вопрос” (объяснение смысла)
4) SOS “нужны примеры” (список ситуаций)
5) благодарность → история
6) чувства → история
7) 10 шаг → сохранение
